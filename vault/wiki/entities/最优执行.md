# 最优执行

> Optimal Execution — **大订单如何拆分成多个小订单**以最小化总交易成本（市场冲击 + 时间风险）的算法

## 一句话定义

机构投资者要执行大订单（如卖出 100 万股），如果一次性下单会**冲击市场价格**（推升或砸盘）；如果拆得太散又承担**价格漂移风险**。最优执行算法在**冲击成本** vs **时间风险**之间求最优拆分轨迹。

## 两阶段框架（本书 6.1）

### 第一阶段：天/周维度的拆分
- 把大订单拆成多日子订单
- 模型：[[Almgren-Chriss模型]]、Bertsimas-Lo 1998
- 输出：每日应执行的份额

### 第二阶段：秒维度的最优配置
- 当日内子订单进一步拆分到秒级
- 模型：基于 [[限价订单簿LOB]] 的供求推断
- 输出：每秒下单的限价/市价、价格、份额

## 核心模型：Almgren-Chriss 2001（行业事实标准）

### 关键假设
- 价格遵循**算术布朗运动**
- 执行产生**永久冲击**（线性）+ **暂时冲击**（线性）
- 投资者有风险厌恶（mean-variance utility）

### 优化问题
```
min     E[Cost] + λ · Var[Cost]
s.t.    Σ x_t = X   （总执行量）
        x_t ≥ 0     （仅卖出）
```

### 最优解形式
- **指数衰减拆分**：早期多卖、后期少卖
- 风险厌恶 λ ↑ → 拆分更急（赶时间）
- 永久冲击系数 ↑ → 拆分更慢（避免推升价格）

### 衍生工具
- **VWAP 算法**：按历史成交量分布拆单
- **TWAP 算法**：按时间均匀拆单（最简单）
- **POV 算法**（Percentage of Volume）：按当下成交量比例下单

| 论文 | 贡献 |

## 三种实操算法

### 1. TWAP（Time-Weighted Average Price）
- 把订单等分成 N 份，每隔 T/N 时间下一份
- **优点**：简单、不依赖预测
- **缺点**：在量小时段下单 → 冲击大

### 2. VWAP（Volume-Weighted Average Price）
- 按历史日内成交量分布拆单（开盘 + 尾盘量大 → 多下）
- **优点**：跟随市场流动性
- **缺点**：依赖历史 volume profile（行情突变时失效）

### 3. POV / Implementation Shortfall
- 按当下实时成交量比例下单（如取 5%-15% of market volume）
- **优点**：动态适应
- **缺点**：成交完成时间不确定

## 实战考量（业界常见）

| 维度 | 典型设置 |
|---|---|
| 风险厌恶系数 λ | 0.01-1 之间，取决于 trader 风险偏好 |
| 永久冲击 b | 估计 1-5 bps per % daily volume |
| 暂时冲击 a | 估计 5-20 bps per % daily volume |
| Volume profile | 60 日历史平均（U 型曲线）|
| 完成时长 | 通常 < 1 个交易日 |

## 与已有素材的关联

### 与 [[2026-05-05-qpb-etf-intraday-momentum]]（ETF 日内动量）
- 该策略是**信号驱动**的入场信号 → 但**未讨论如何拆单**
- 实盘时需要叠加最优执行（VWAP / TWAP）控制冲击成本

### 与 [[多信号策略]]
- 浙商证券系列二讨论"何时入场"
- 但**进场后如何拆单**是另一个问题，本主题填补

### 与 V25 项目
- V25 目前用**收盘价回测**（隐含 TWAP 或单点执行）
- 实盘部署时必须加入最优执行模块（特别是 ETF 池有大订单时）

## 优势 / 劣势

✅ 业界事实标准，券商执行系统通常内置
✅ 数学清晰可证（Mean-variance utility）
✅ 与 [[限价订单簿LOB]] 数据天然结合

⚠ 永久 / 暂时冲击系数**估计困难**（需要 tick 级历史数据）
⚠ 极端行情下（fat tails）布朗运动假设崩溃
⚠ 不考虑交易对手博弈（HFT 可能 front-running）
